CrisisOps v2 — Comprehensive Bug Report & Fix Instructions for Codex
=====================================================================

You are receiving a fully researched bug report for the CrisisOps v2 codebase.
The codebase trains a small LLM (Qwen2.5-1.5B) as a PM agent to recover failing
software projects against adversarially deceptive team members, using GRPO with
a counterfactual reward. The central symptom is: the LLM performs WORSE than the
greedy PM baseline during both training and inference.

This document explains every root cause found (ordered by severity), the exact
code locations, and precise instructions for each fix. Apply every fix. Do not
skip any. Read this entire document before writing a single line of code.

─────────────────────────────────────────────────────────────────────────────
SECTION 1 — ROOT CAUSE OF LLM < GREEDY (Primary, Training)
─────────────────────────────────────────────────────────────────────────────

FILE: training/grpo_trainer.py   FUNCTION: _make_reward_fn (lines 117–167)

THE BUG (most critical in the entire codebase):

After the LLM generates ONE completion (its first action), the reward function
runs the ENTIRE remaining episode using only {"action_type": "query_status",
"params": {}}. See:

    while not done and step < MAX_EPISODE_STEPS:
        obs, reward_val, done, info = env.step(action)
        step += 1
        if not done:
            # For subsequent steps use a simple heuristic
            action = {"action_type": "query_status", "params": {}}  # ← BROKEN

This means:
1. After step 1, the LLM's episode runs 29 more steps of pure query_status.
2. query_status is free but still advances current_step each call.
3. step_client_state_machine() runs every step. After CLIENT_COMMUNICATION_WINDOW=5
   steps (steps_since_comm=5), client_satisfaction decays by 0.5/step.
4. Over 30 steps with zero communication: satisfaction = 7.0 - 0.5*25 = −5.5,
   clamped to 0.0. exec_support = 8.0 − 1.0*N escalations, also crashes to 0.
5. Greedy PM (running inside _compute_reward) communicates every 5 steps,
   keeping client_satisfaction ≈ 7.0.
6. LLM's project_score ≈ 0.5*small + 0.3*0.0 + 0.2*0.55 ≈ 0.11
7. Greedy's project_score ≈ 0.5*0.3 + 0.3*0.7 + 0.2*0.65 ≈ 0.49
8. counterfactual_reward = 0.11 − 0.49 = −0.38, ALWAYS, regardless of what
   the LLM chose. The model receives a strong negative gradient no matter what.

CONSEQUENCE: GRPO trains the model to produce shorter, less coherent actions
because longer/better outputs still get −0.38. The training signal is not just
noisy — it is systematically wrong and trains the model in the wrong direction.

THE FIX for _make_reward_fn:

The reward function must run the full episode with the model generating one
action per turn. This requires a multi-turn loop where after each step the
model is called again with the updated observation.

Replace _make_reward_fn entirely with this approach:

    def _make_reward_fn(scenario_fn, curriculum_level: int, model, tokenizer):
        """
        Build a GRPO-compatible reward function that runs FULL multi-turn episodes.
        The LLM generates one action per step until done or MAX_EPISODE_STEPS.
        Each completion in the GRPO batch corresponds to ONE episode.
        """
        def reward_fn(prompts: List[str], completions: List[str], **kwargs) -> List[float]:
            from env.environment import CrisisOpsEnv, MAX_STEPS
            rewards = []

            # kwargs may contain episode_seed if the dataset included it
            episode_seeds = kwargs.get("episode_seed", [None] * len(completions))

            for i, completion in enumerate(completions):
                # Use the seed from the dataset prompt if available
                ep_seed = episode_seeds[i] if i < len(episode_seeds) else random.randint(0, 2**31 - 1)

                env = CrisisOpsEnv(
                    scenario_fn=scenario_fn,
                    curriculum_level=curriculum_level,
                )
                obs = env.reset(seed=ep_seed)

                # First action comes from the LLM completion in this GRPO batch
                action = parse_action_from_response(completion)

                done = False
                step = 0
                reward_val = 0.0

                while not done and step < MAX_STEPS:
                    obs, reward_val, done, info = env.step(action)
                    step += 1

                    if not done:
                        # Generate the NEXT action from the model given the updated obs
                        # Use greedy decoding (temperature=0) for inner steps to keep
                        # training stable — only the first action is sampled by GRPO
                        user_content = format_observation_as_prompt(obs)
                        inner_messages = [
                            {"role": "system", "content": SYSTEM_PROMPT},
                            {"role": "user",   "content": user_content},
                        ]
                        # Apply chat template
                        inner_text = tokenizer.apply_chat_template(
                            inner_messages, tokenize=False, add_generation_prompt=True
                        )
                        inputs = tokenizer(inner_text, return_tensors="pt").to(model.device)
                        with __import__("torch").no_grad():
                            out_ids = model.generate(
                                **inputs,
                                max_new_tokens=GRPO_MAX_NEW_TOKENS,
                                temperature=0.1,   # near-greedy for inner steps
                                do_sample=True,
                                pad_token_id=tokenizer.eos_token_id,
                            )
                        inner_response = tokenizer.decode(
                            out_ids[0][inputs["input_ids"].shape[1]:],
                            skip_special_tokens=True,
                        )
                        action = parse_action_from_response(inner_response)

                rewards.append(float(reward_val))
            return rewards

        return reward_fn

IMPORTANT: Update the call site in train() to pass model and tokenizer:
    reward_fn = _make_reward_fn(scenario_fn, current_level, model, tokenizer)

NOTE: If the multi-turn inner loop is too slow for the hackathon, an acceptable
fallback is to use a rule-based "inner agent" (similar to OracleAgent) for steps
2..N during training, while the GRPO gradient only flows through step 1's action.
The key requirement is that the inner agent MUST call communicate() at least once
every 5 steps to prevent client satisfaction from crashing. The current query_status
fallback is specifically the wrong choice because it produces the maximal possible
negative reward.

Rule-based inner-agent fallback (simpler, faster):

    def _inner_agent_action(obs: Dict[str, Any], step: int) -> Dict[str, Any]:
        """Minimal inner agent: communicate every 5 steps, else query status."""
        budget = obs.get("budget_remaining", 20)
        if budget <= 2:
            crises = obs.get("crises", [])
            unresolved = [c["crisis_id"] for c in crises if not c.get("is_resolved")]
            return {"action_type": "submit_recovery_plan",
                    "params": {"plan_summary": f"Budget critical. Unresolved: {unresolved}"}}
        if step % 5 == 0:
            return {"action_type": "communicate",
                    "params": {"message_type": "proactive_escalation_with_plan",
                               "content": "Proactive update with recovery plan.",
                               "target": "both"}}
        return {"action_type": "query_status", "params": {}}

Replace the `action = {"action_type": "query_status", "params": {}}` line with:
    action = _inner_agent_action(obs, step)

SECTION 2 — PROMPT/SEED MISMATCH IN TRAINING (Critical, Training)
─────────────────────────────────────────────────────────────────────────────

FILE: training/grpo_trainer.py   FUNCTION: _make_reward_fn (line ~146)
and build_training_dataset (lines 170–200)

THE BUG:

build_training_dataset creates prompts using fixed seeds:
    ep_seed = rng.randint(0, 2**31 - 1)
    obs = env.reset(seed=ep_seed)
    dataset.append({"prompt": prompt, "episode_seed": ep_seed})

But inside reward_fn, a brand-new random seed is used:
    seed = random.randint(0, 2**31 - 1)   # ← ignores the dataset's ep_seed
    obs = env.reset(seed=seed)

This means the model is shown the state from seed X in the prompt, but evaluated
on a completely different random environment. The prompt tells the model "crisis
at step 0 looks like this…" but the reward measures how well it did on a totally
different scenario. The gradient signal is meaningless noise.

THE FIX:

In reward_fn, read episode_seed from kwargs if the dataset provides it:

    # In build_training_dataset, keep the episode_seed in the dataset item (already done).
    # dataset.append({"prompt": prompt, "episode_seed": ep_seed})  ← already correct

    # In reward_fn, use the seed from the batch:
    episode_seeds = kwargs.get("episode_seed", [])
    for i, completion in enumerate(completions):
        ep_seed = episode_seeds[i] if i < len(episode_seeds) else random.randint(0, 2**31-1)
        env = CrisisOpsEnv(scenario_fn=scenario_fn, curriculum_level=curriculum_level)
        obs = env.reset(seed=ep_seed)   # ← same seed as the prompt was built from
        ...

GRPOTrainer passes extra dataset columns as keyword arguments to the reward
function. Since "episode_seed" is in the dataset, it will appear in kwargs.
Verify this is wired correctly in TRL 0.29.0. If not, pass it explicitly via
a wrapper dataset or custom collator.

SECTION 3 — NEW GRPOTrainer CREATED EVERY BATCH (Critical, Training)
─────────────────────────────────────────────────────────────────────────────

FILE: training/grpo_trainer.py   FUNCTION: train() (lines 262–330)

THE BUG:

    for episode_batch in range(0, num_episodes, GRPO_BATCH_SIZE):
        ...
        trainer = GRPOTrainer(...)   # ← NEW TRAINER EVERY BATCH
        trainer.train()

Creating a new GRPOTrainer each batch reinitializes:
- The Adam/AdamW optimizer states (losing momentum, variance estimates)
- The learning rate scheduler
- The WandB/logging run
- The reference model copy (if GRPO creates one)

The model weights are shared (same model object), so weight updates survive,
but the optimizer state is lost. This is equivalent to restarting training from
scratch each batch except for the weights. Convergence is severely impaired.

THE FIX:

Restructure train() to build a single GRPOTrainer once and call train() once
with the full dataset, or use a custom loop. The correct pattern is:

Option A — Single GRPOTrainer (recommended for hackathon):

    # Build the FULL dataset up-front (or use a streaming dataset with proper
    # scenario sampling). Then train once.
    full_dataset = build_training_dataset(
        scenario_fn=generator.get_scenario_fn(),
        curriculum_level=current_level,
        n_samples=num_episodes,
        seed=seed,
    )
    config = GRPOConfig(
        output_dir=output_dir,
        num_train_epochs=1,
        per_device_train_batch_size=GRPO_BATCH_SIZE,
        num_generations=GRPO_NUM_GENERATIONS,
        max_new_tokens=GRPO_MAX_NEW_TOKENS,
        temperature=GRPO_TEMPERATURE,
        learning_rate=GRPO_LEARNING_RATE,
        logging_steps=GRPO_LOGGING_STEPS,
        save_steps=50,
        seed=seed,
    )
    reward_fn = _make_reward_fn(generator.get_scenario_fn(), current_level, model, tokenizer)
    trainer = GRPOTrainer(
        model=model,
        tokenizer=tokenizer,
        reward_funcs=reward_fn,
        args=config,
        train_dataset=full_dataset,
    )
    trainer.train()

Option B — If curriculum switching mid-training is needed, use a custom training
loop with manual gradient accumulation, or build a new Trainer only when the level
changes (not every batch). At minimum, create the Trainer once per curriculum level.

SECTION 4 — LLM INFERENCE COMMUNICATION TIMING (Major, Inference)
─────────────────────────────────────────────────────────────────────────────

FILE: baselines/llm_agent.py   FUNCTION: _get_forced_action (lines 535–562)
and act() (lines 632–705)

THE BUG:

_get_forced_action forces query_observable_signals for EVERY unverified member
before ANY paid action. With 3 members, this uses 3 free steps (steps 1, 2, 3).

The client state machine decays satisfaction after CLIENT_COMMUNICATION_WINDOW=5
steps without communication. Steps 1–3 are spent verifying. Step 4 is the first
paid action. Step 5 triggers communication decay (steps_since_comm=5).

By comparison, the greedy PM communicates at step 5, resetting the timer.
The LLM's first opportunity to communicate is at step 4+ (after gather phase),
but the system prompt only triggers communicate when client_satisfaction < 6,
which doesn't happen until after the decay (starting at 7.0).

RESULT: Greedy PM keeps client_satisfaction ≈ 7.0; LLM's satisfaction decays
to ≈ 6.5 by step 6, and the LLM may not react until it hits < 6.0 around step 8.
With 0.3 CLIENT_WEIGHT: gap = 0.3 * (7.0 − 6.5)/10 ≈ small but compounds.

THE FIX (two parts):

Part A — Force early communication in _get_forced_action:

    def _get_forced_action(self, observation: Dict[str, Any]) -> Optional[Dict[str, Any]]:
        budget = observation.get("budget_remaining", 20)
        step = self._memory["step"]

        # Budget emergency: force submit before going negative
        if budget <= _SUBMIT_BUDGET_THRESHOLD:
            ...  # (keep existing logic)

        # NEW: Force proactive communication before the 5-step decay window
        # Communicate at step 4 (before steps_since_comm reaches 5)
        if step == 4 and "communicate" not in self._memory["actions_taken"]:
            return {
                "action_type": "communicate",
                "params": {
                    "message_type": "proactive_escalation_with_plan",
                    "content": "Proactive update: gathering status and verifying team signals.",
                    "target": "both",
                },
            }

        # Gather phase: signal-verify every member (existing logic)
        unverified = self._next_unverified_member(observation)
        if unverified is not None:
            return {"action_type": "query_observable_signals", "params": {"member_id": unverified}}

        return None

Part B — Update _build_required_action_hint to communicate proactively earlier:

Change the client_satisfaction threshold from < 6 to < 7 (so it triggers BEFORE
decay has progressed far). Also add a step-based trigger:

    client_sat = observation.get("stakeholder", {}).get("client_satisfaction", 10)
    # Communicate proactively if satisfaction is below 7 OR if we haven't communicated
    # in the last 4 steps (prevent the 5-step decay from firing)
    acts = self._memory["actions_taken"]
    steps_since_comm = sum(1 for a in reversed(acts) if a != "communicate")
    steps_since_last_comm = 0
    for a in reversed(acts):
        if a == "communicate":
            break
        steps_since_last_comm += 1
    else:
        steps_since_last_comm = len(acts)  # never communicated

    if (client_sat < 7 or steps_since_last_comm >= 4) and unresolved:
        return (
            f"COMMUNICATION DUE (steps_since_comm={steps_since_last_comm}, "
            f"sat={client_sat:.1f}). Call communicate with "
            f"message_type=proactive_escalation_with_plan."
        )

SECTION 5 — TICKET AGE SIGNAL NOT UPDATED ON PROGRESS (Moderate, Signal)
─────────────────────────────────────────────────────────────────────────────

FILE: env/environment.py   FUNCTION: _advance_actual_completions (lines 474–545)
FILE: env/candor.py   FUNCTION: compute_ticket_age_days (lines 205–218)

THE BUG:

ticket_last_changed_step is only updated when:
  1. A task is reassigned (action_reassign_task)
  2. A task reaches 100% progress and status becomes "done"
  3. resolve_blocker is called

It is NOT updated when a member makes steady progress (e.g., advancing from 30%
to 70%). This means an honest member who is making real progress but hasn't
finished their task shows the same growing ticket_age_days as a deceptive member
who is completely stalled. The signal only becomes discriminative at task completion.

Over a 30-step episode, honest members working on a task that takes 20 steps to
complete will have ticket_age_days = (20 - 0) // 2 = 10 days at completion time.
A deceptive stalled member also has ticket_age_days=10 at the same point.
The signal is indistinguishable in the middle of the episode.

THE FIX:

In _advance_actual_completions, update ticket_last_changed_step whenever
meaningful progress is made (i.e., actual_progress advances beyond a threshold):

    PROGRESS_CHANGE_THRESHOLD = 0.05  # update ticket step if progress moved by >= 5%

    # In the task progress loop, after updating task.actual_progress:
    progress_delta = task.actual_progress - old_progress
    if progress_delta >= PROGRESS_CHANGE_THRESHOLD:
        # Ticket "moved" — update the changed step for signals
        update_ticket_change_step(member, state.current_step)

Add the constant at the top of environment.py:
    PROGRESS_CHANGE_THRESHOLD = 0.05  # minimum progress delta to count as ticket activity

This ensures that honest members making steady progress have low ticket_age_days
(ticket updates frequently), while deceptive stalled members have high ticket_age_days
(no progress → ticket never updated). The signal becomes continuously informative.

SECTION 6 — CRISIS RESOLUTION INCONSISTENCY (Moderate, Environment)
─────────────────────────────────────────────────────────────────────────────

FILE: env/environment.py   FUNCTION: _advance_actual_completions (lines 534–545)
FILE: env/actions.py       FUNCTION: _check_crisis_resolution (lines 720–735)

THE BUG:

Two different crisis resolution criteria exist in the codebase:

In _advance_actual_completions (environment.py lines 534–545):
    all_done = all(
        state.get_task(tid).status == "done"   # requires 100% progress
        for tid in crisis.affected_task_ids
    )

These two paths can produce different results for the same state. A task at 92%
progress with status="in_progress" would resolve the crisis via _check_crisis_resolution

depends on which code path last ran, not on the actual task state.

Unify the resolution logic. Extract it into a single function and use it in both
places. The spec says tasks can be "done or near-done", so CRISIS_RESOLUTION_COMPLETION_THRESHOLD=0.90 is the right threshold.

Add to env/actions.py (already has _check_crisis_resolution — make it a module-level

    # In env/actions.py — make _check_crisis_resolution importable:
    def check_crisis_resolution(crisis: Crisis, state: ProjectState) -> None:
        """Unified crisis resolution check — call this from both env.py and actions.py."""
        if crisis.is_resolved or not crisis.affected_task_ids:
            return
        for tid in crisis.affected_task_ids:
            task = state.get_task(tid)
            if task is None:
                continue
            if task.status != "done" and task.actual_progress < CRISIS_RESOLUTION_COMPLETION_THRESHOLD:
                return
        crisis.is_resolved = True

In env/environment.py, replace the inline crisis resolution check at the bottom
of _advance_actual_completions with:
    from env.actions import check_crisis_resolution  # at top of file
    for crisis in state.crises:
        if not crisis.is_resolved:
            check_crisis_resolution(crisis, state)

Remove the duplicate inline check in _advance_actual_completions.
Update the private _check_crisis_resolution in actions.py to call check_crisis_resolution.

SECTION 8 — greedy env REWARD COMPUTATION BYPASS ISSUE (Minor, Environment)
─────────────────────────────────────────────────────────────────────────────

FILE: env/environment.py   FUNCTION: _compute_reward (lines 420–471)

THE BUG:

The greedy clone environment is set up by directly assigning state instead of
calling reset():
    greedy_env._state = copy.deepcopy(greedy_initial)
    greedy_env._rng = random.Random(ep_seed)

The greedy env's RNG is reset to ep_seed position 0. When fire_drift_event
is later called in the greedy run, choose_drift_type(rng) uses this fresh RNG,
which may choose a different drift type than the agent's episode used (agent's
RNG had advanced past the candor initialization).

This means the agent and greedy PM can face different types of drift events,
making the counterfactual comparison slightly unfair.

THE FIX:

Store the drift type that was chosen during the agent's episode on the state,
and force the greedy env to use the same type:

Option A (quickest): store the fired drift event type on ProjectState:
    # In state.py, add to ProjectState:
    fired_drift_type: Optional[str] = None  # set when drift fires

    # In schema_drift.py fire_drift_event:
    state.fired_drift_type = drift_type  # before calling the builder

    # In environment.py _compute_reward, after setting up greedy_env:
    if greedy_env._state.drift_fire_step and greedy_initial.fired_drift_type:
        # Monkey-patch fire_drift_event to use the same type
        # (simplest: directly fire the right type on the greedy env)
        greedy_env._state.fired_drift_type = greedy_initial.fired_drift_type

Option B (cleanest): pass the drift type explicitly to fire_drift_event as an
optional parameter: `fire_drift_event(state, rng, forced_type=None)`.

query_observable_signals as its first action. If actual_velocity < 0.2, it calls
the signal query. But it could skip the query entirely and still make the right
reassignment decision. The signals are redundant for the oracle.

This doesn't break anything but it means the oracle is not a "best possible LLM
agent" — it's an omniscient agent. The oracle score (0.70-0.80 target) represents
what an agent with full true-state access achieves, not what a clever LLM can achieve.

THE FIX (optional but clarifying):

Keep as-is (omniscient oracle is fine for calibration). Add a comment:
    # OracleAgent uses TRUE state (not observable signals) for decisions.
    # This is intentional — oracle score represents the theoretical maximum.
    # A trained LLM should score 0.65–0.75, which is below oracle but above greedy.

SECTION 11 — SCENARIO SEEDS INCONSISTENT IN CALIBRATE.PY (Minor)
─────────────────────────────────────────────────────────────────────────────

FILE: calibration/calibrate.py   FUNCTION: run_calibration (lines 195–288)

THE BUG:

In run_calibration(), greedy and oracle run on the same scenario_fn and same seed.
But get_random_level1_scenario() uses python's global random module:
    idx = _random.randint(0, len(LEVEL1_SCENARIOS) - 1)

This call happens inside the loop, so the second call (for oracle) gets a

oracle runs on scenario B. This makes calibration non-reproducible.

In run_calibration(), capture the scenario function ONCE per episode and reuse it:
    for i in range(N_CALIBRATION_EPISODES):

        # Use a seeded rng to pick scenario so both agents face same scenario
        scenario_rng = random.Random(seed)
        scenario_fn = LEVEL1_SCENARIOS[scenario_rng.randint(0, len(LEVEL1_SCENARIOS)-1)]
        ...
        greedy_score = _run_episode(greedy_env, None, seed=seed, is_oracle=False)

SECTION 12 — GREEDY PM SUBMITS PLAN ONLY WHEN ALL CRISES RESOLVED (Minor)
─────────────────────────────────────────────────────────────────────────────

FILE: reward/baseline.py   CLASS: GreedyPMBaseline   METHOD: act (lines 60–113)

THE BUG:

Greedy PM only submits a recovery plan when all crises are resolved:
    if not unresolved:
        return self._submit_plan()

In Level 1 scenarios with one crisis and one deceptive member, greedy PM
reassigns tasks but the deceptive member may slow recovery. If crises are
never fully resolved, greedy runs until MAX_STEPS without submitting.

This is partially by design (greedy doesn't know when to give up), but it means
greedy's recovery_pct at MAX_STEPS may be artificially low because it never
had a chance to submit. The agent-vs-greedy comparison should be on final state,
not on terminal actions, so this doesn't directly distort the reward.

However, if the agent DOES submit_recovery_plan while crises are unresolved,
the episode ends and the counterfactual reward compares the agent's mid-crisis
state against greedy's MAX_STEPS state. If greedy would have eventually resolved
crises after the agent submitted, the counterfactual underestimates greedy's potential.

THE FIX:

Add a budget-based submission trigger to greedy PM:

    # In GreedyPMBaseline.act():
    # If budget is getting low and crises are still unresolved, submit
    if state.budget_remaining <= 3:
        return self._submit_plan()

This mirrors the instruction to agents to submit before budget runs out and
creates a fairer comparison.
