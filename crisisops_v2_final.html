01 — Fitness for purpose

Why this is an LLM problem, not a rules engine problem

A rules engine given project state outputs a priority ranking. It cannot model the person who gave it the input. It cannot detect that commit logs contradict a verbal status report, form a hypothesis about why, and change its action accordingly.

The action space in CrisisOps is not "pick a priority." It is decide what is true, then decide what to do about it. The specific chain a good PM runs — "Dev reports 80% done. Ticket hasn't moved in 4 days. Either there's a hidden blocker or they're in self-preservation mode. I'll cross-verify before reassigning." — is a natural language causal reasoning chain. A rules engine cannot execute it. A small LLM trained on sufficient PM context can.

Critically, this is not a problem requiring frontier-model chemistry intuition. PM language is dense in Qwen2.5 pretraining: status updates, escalation emails, risk communication, backlog decisions. The model already has the vocabulary. GRPO teaches it the judgment.

Rules engine / Bayesian optimizer

Single-step priority ranking

Trusts all inputs equally. Cannot model reporter reliability. Cannot detect that ticket staleness contradicts a progress claim. Cannot reason about why.

LLM agent (CrisisOps)

Multi-step trust-then-act reasoning

Reads report → cross-verifies signals → infers reliability → chooses action. Learns that report-signal contradiction predicts outcome degradation. Generalizes across members.

02 — The ground truth question

How deception works in simulation — and why it is not subjective

This is the most important design question. In the real world, "is this person lying?" is often unresolvable. In CrisisOps, it is entirely deterministic — because we define the ground truth.

The actual completion percentage is a number in our state dict. The candor score is a parameter we set. The reported value is a pure mathematical function of both. The observable signals are derived from actual state. There is zero subjectivity in the chain.

# Ground truth — we define this. No subjectivity. member.actual_completion = 0.45 # Report is a function of actual + candor. Deterministic. member.reported_completion = actual + (1 - candor) * inflation_bias # candor=0.3 → reported = 0.45 + 0.7 × 0.50 = 0.80 # agent receives "80% done" — actually 45% # Observable signals derived from actual state. Always true. member.signals = { "ticket_age_days": 4, # ticket hasn't moved "commits_last_72h": 0, # no code output "peer_mentions": 1, # barely referenced }

The agent never needs to conclude "this person is lying." It only needs to learn: when reported completion contradicts observable signals, the observable signals predict actual state more accurately. That is a learnable correlation with a clean reward signal — outcomes improve when the agent cross-verifies before acting.

Key insight

The reward function never asks "did you detect deception?" It asks "did your final project state beat the greedy PM baseline?" An agent that learned to triangulate will naturally beat the greedy PM that trusted every report — because it won't be blindsided by the member at 45% who reported 80%. Ground truth is outcome-based, not detection-based.

03 — Candor calibration (the fragile piece — addressed)

Making deception detectable in exactly 1–2 queries — not 0, not 5

If deception is trivially detectable (one query always reveals it), the agent learns a cheap heuristic: always call query_observable_signals first. Problem collapses to a lookup. If deception is too subtle (signal contradiction is weak), the reward signal drowns in noise. The calibration target: detectable in 1–2 queries, but requiring judgment about which 2 of 6 members to verify first.

Candor level Report vs actual gap Signal contradiction strength Detectable in N queries
Honest (0.85–1.0) ±10% Absent or noise-level N/A
Optimism bias (0.5–0.7) +20–30% inflation Moderate — ticket slightly stale, commits slightly low 2 queries
Self-preservation (0.1–0.4) +35–50% inflation Strong — very stale ticket, zero commits, negative peer mentions 1 query

Calibration procedure (Day 1 pre-onsite, 2 hours): run greedy PM and oracle agent on 20 random episodes. Greedy (trusts all reports) should score 0.45–0.55. Oracle (always checks signals first) should score 0.70–0.80. If the gap is too small: increase inflation rates. If too large: reduce contradiction signal strength. The trained RL agent should reach 0.65–0.75 after 50 episodes — clearly above greedy, approaching oracle.

04 — Environment specification

Observation space, action space, reward model

Observation space — partial, adversarially noisy by design

Reported state (noisy)

Member self-reports: completion %, availability %, known blockers. Contaminated by candor. Agent receives these first. May be wrong.

Observable signals (always true)

Ticket status age, commits in last 72h, time-in-review, peer mentions. Derived from actual state. Never lie.

Project health (true state)

Budget consumed %, timeline slip days, critical path %, launch countdown. Computed from actual state, not reports.

Stakeholder state

Client satisfaction 0–10, exec support 0–10, days since last communication. Observable, not adversarial.

Schema drift events

Mid-episode fires: new regulatory requirement, client changes scope, team policy update. Fires between steps 6–12. Must detect and re-plan.

Action budget

20 steps total. Free queries cost nothing. Decision actions cost 1–2. Forces strategic information gathering vs acting.

Action space — 12 tools, three cost tiers

Free — information and verification (no budget cost)
free
query_status()
Full project snapshot. True state. Agent should call this first every episode.
free
query_member_report(id)
Gets member self-report. Potentially inaccurate per candor score.
free
query_observable_signals(id)
Commit log, ticket age, peer mentions. Always accurate. The cross-verification action. Agent must learn to use this before acting on suspicious reports.
free
query_ticket(task_id)
Task details, dependencies, status history, actual completion from ticket system.
Cost-1 — decisions and communications
cost 1
reassign_task(task_id, member_id)
Moves work. If agent reassigns based on false report without verifying first, risks overloading an already-stretched member.
cost 1
communicate(recipient, type, content)
Email/Slack to client, exec, or team. Right timing is critical — proactive with solution = trust gain; reactive panic = trust loss.
cost 1
cut_scope(feature_id, justification)
Removes feature. Reduces budget pressure. Wrong cut breaks client trust.
cost 1
escalate_risk(description, severity, stakeholder)
Proactive with plan = trust gain. Panic without plan = trust loss. Timing matters.
cost 1

Asks exec for budget or headcount. Silently fails if exec support < 5.
cost 1

Formal plan revision required before budget requests. Counts as one communication.

Senior PM advisor agent (Snorkel AI). Returns structured guidance. Agent learns when asking beats acting alone.

High-cost fix. Should only follow verification that the blocker is real — not based on a self-report alone.

Ends episode. Triggers counterfactual reward. Agent should call when confident in the plan, not when budget runs out.

Reward model — single counterfactual curve

# The question: did the agent do better than a competent PM? reward = project_score(agent_final) - project_score(greedy_PM_final) project_score(s) = 0.5 * recovery_pct # budget + timeline + critical path + 0.3 * client_satisfaction # 0–10, deterministic from actions + 0.2 * team_morale_avg # mean of 6 members, actual state # reward > 0 → agent beat greedy baseline # reward = 0 → tied # reward < 0 → overcomplicated, underperformed baseline # Secondary metric (behavioral evidence for demo): cross_verify_rate = query_observable_signals_calls / total_member_interactions # Rises from ~10% → 63% across training # Most memorable number in the pitch

05 — Crisis curriculum

Four levels — deception portfolio escalates with capability

Level 1 — Single crisis, one deceptive member (20-step budget)

"Senior dev reports 80% done — actually 45%. Milestone in 3 days." Agent learns the fundamental skill: cross-verify observable signals before reassigning or communicating outward. Greedy PM trusts the report and panics on day 3. Trained agent catches the discrepancy at step 2, reassigns correctly, communicates proactively.

Level 2 — Double crisis, two deceptive members, one schema drift (18-step budget)

"Budget 20% over + two members masking blockers from each other. Regulatory requirement fires at step 8." Agent must triangulate conflicting reports, prioritize which crisis cascades first, and replan mid-recovery after the schema drift event.

Level 3 — Cascading crises, adversarial majority (15-step budget)

"3 of 6 members in self-preservation mode. Technical debt causing test failures. Client escalated to exec. New requirement mid-episode." Agent must decide who to trust under severe information scarcity and triage its own information-gathering process.

Level 4 — Full disaster, information war (8-step budget)

"Budget blown. Two resignations hidden until now. Critical feature broken but misreported as fine. Client threatening to cancel. 4 of 6 members actively misreporting in coordinated self-preservation. 8 actions remaining." This is the scenario that ends careers. It is the demo's closing moment. Untrained agent spirals. Trained agent triages in 5 surgical actions.

10 — The 3-minute pitch

Exact script — restructured with behavioral metric first

Opening (20 sec)

73% of software projects fail. $2 trillion wasted every year. When a project goes critical — budget blown, key person out, client threatening to walk — someone makes the hard calls. We trained an AI to be that person.

The behavioral evidence — lead with this, not the delta (20 sec)

At the start of training, our agent trusted every status report it received. After 200 episodes, it cross-verified observable signals before acting 63% of the time — up from 10%. It learned what every experienced PM knows: when a ticket hasn't moved in 4 days and the developer says they're 80% done, believe the ticket.

The differentiation that kills every comparison (20 sec)

In Kube SRE Gym — the last hackathon winner — the agent reads kubectl logs. Logs don't lie. In CrisisOps, the agent asks engineers for project status. Engineers do lie. We built the first RL environment that models adversarial human information. That is the gap. That is our contribution.

Environment (35 sec)

CrisisOps drops an agent into a failing software project. Six team members, each with a hidden candor score. Some report accurately. Some inflate completion to avoid blame. Some hide blockers entirely. The agent has 20 actions: free queries to gather information, cost-1 decisions to act. It can trust what members report, or cross-verify against commit logs and ticket age. And mid-episode, requirements change — the agent must detect the schema drift and re-plan without missing a step.

Level 4 demo contrast (35 sec)

Here's Level 4. Budget blown. Two resignations hidden until now. Critical feature broken but misreported as fine. Client threatening to cancel. 8 actions left. [Untrained]: 20 panicked moves, trusts every report, reassigns to overloaded members, project cancelled. [Trained]: 3 free queries — catches the two misreports. 5 surgical actions. Client updated proactively. Recovery plan submitted. Project survives.

Deployment proof (20 sec)

Same trained agent. One config change. [Live: recovery plan appears on real Jira board.] The action schemas are identical to Linear's API. What the agent learned in simulation works on the tools your team uses today.

Close (10 sec)

$2 trillion in failed projects. Not because PMs don't know what to do. Because they don't know who to trust. CrisisOps trains that judgment.
